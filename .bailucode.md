# Repository Guidelines

## 项目结构与模块组织

本项目采用模块化的目录结构，便于维护和扩展：

```
project-root/
├── src/                  # 源代码目录
│   ├── components/       # 可复用组件
│   ├── utils/           # 工具函数
│   ├── services/        # 服务层（API调用等）
│   └── assets/          # 静态资源
│       ├── css/         # 样式文件
│       ├── js/          # JavaScript文件
│       └── images/      # 图片资源
├── tests/               # 测试文件目录
├── docs/                # 项目文档
├── dist/                # 构建输出目录
└── config/              # 配置文件
```

**关键原则：**
- 每个模块职责单一，避免将所有代码写在单一文件中
- 前端项目中，HTML、CSS、JavaScript 必须分离
- 使用文件夹组织相关文件，保持扩展性

## 编码风格与命名规范

### 通用规范

- **缩进：** 使用 2 个空格（不使用 Tab）
- **行尾：** 统一使用 LF（Unix风格）
- **编码：** UTF-8
- **分号：** JavaScript/TypeScript 中使用分号结尾

### 命名约定

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名（组件） | PascalCase | `UserProfile.jsx` |
| 文件名（工具） | camelCase | `formatDate.js` |
| 文件名（样式） | kebab-case | `user-profile.css` |
| 变量/函数 | camelCase | `getUserData()` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 类/组件 | PascalCase | `class DataService` |
| CSS 类名 | kebab-case | `.btn-primary` |

### 代码风格工具

- **Linter:** ESLint（配置见 `.eslintrc.js`）
- **Formatter:** Prettier（配置见 `.prettierrc`）
- 提交前会自动运行 lint 检查（通过 husky hooks）

### 最佳实践

- 使用有意义的变量名，避免单字母变量（除循环索引外）
- 函数应保持简短，单一职责
- 优先使用 `const`，需要重新赋值时使用 `let`，避免 `var`
- 前端美化时使用 FontAwesome 图标或 SVG，避免 emoji
- 组件设计应现代化，注重用户体验

## 测试指南

### 测试框架

- **单元测试：** Jest / Vitest
- **E2E 测试：** Cypress / Playwright（如适用）
- **组件测试：** React Testing Library / Vue Test Utils

### 测试覆盖率要求

- 核心业务逻辑：≥ 80%
- 工具函数：≥ 90%
- UI 组件：≥ 70%

### 测试文件命名

```
src/utils/formatDate.js
tests/utils/formatDate.test.js

src/components/UserProfile.jsx
tests/components/UserProfile.spec.jsx
```

### 测试规范

- 每个测试用例应独立，不依赖其他测试
- 使用 `describe` 分组相关测试
- 测试描述应清晰表达测试意图
- Mock 外部依赖（API、数据库等）

**示例：**

```javascript
describe('formatDate', () => {
  it('应该正确格式化日期为 YYYY-MM-DD 格式', () => {
    const date = new Date('2024-01-15');
    expect(formatDate(date)).toBe('2024-01-15');
  });

  it('应该处理无效日期输入', () => {
    expect(formatDate(null)).toBe('Invalid Date');
  });
});
```

## 提交与 Pull Request 规范

### Commit Message 格式

采用约定式提交（Conventional Commits）规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）：**
- `feat`: 新功能
- `fix`: 修复 Bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 重构（既非新功能也非修复）
- `perf`: 性能优化
- `test`: 添加或修改测试
- `chore`: 构建工具或辅助工具的变动

**示例：**

```
feat(auth): 添加用户登录功能

实现了基于 JWT 的用户认证系统，包括：
- 登录接口
- Token 验证中间件
- 刷新 Token 机制

Closes #123
```

### Pull Request 要求

1. **标题：** 简洁明了，说明主要变更
2. **描述：** 必须包含以下内容：
   - 变更的目的和背景
   - 实现的具体方案
   - 测试方法和结果
   - 截图或 GIF（UI 变更时必需）
3. **关联 Issue：** 使用 `Closes #issue-number` 或 `Refs #issue-number`
4. **审查清单：**
   - [ ] 代码已通过所有测试
   - [ ] 已添加必要的测试用例
   - [ ] 文档已更新（如适用）
   - [ ] 无 Lint 警告或错误
   - [ ] 已在本地测试功能

### 分支策略

- `main`：生产环境稳定分支
- `develop`：开发分支
- `feature/*`：功能开发分支
- `fix/*`：Bug 修复分支
- `hotfix/*`：紧急修复分支

**工作流程：**

```bash
# 创建功能分支
git checkout -b feature/user-authentication

# 开发完成后提交
git add .
git commit -m "feat(auth): 添加用户登录功能"

# 推送并创建 PR
git push origin feature/user-authentication
```
